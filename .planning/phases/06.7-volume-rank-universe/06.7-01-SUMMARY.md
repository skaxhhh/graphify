### Auth Analysis

The LOGOUT response from `getJsonData.cmd` is a **server-side session check failure**, not a network block. The GenerateOTP step succeeds (accessible), but `getJsonData.cmd` requires a valid KRX login session cookie (KRX_ID + KRX_PW authentication). This is consistent with RESEARCH findings that KRX tightened unauthenticated access in 2024.

Three options:
1. **Implement KRX_ID/KRX_PW session login** — pykrx 1.2.8 flow (login → session cookie → getJsonData.cmd). Requires KRX account credentials as env vars.
2. **Yahoo Finance 5m intraday cumulative fallback** — per-symbol Yahoo Finance polling, accumulate volume, rank. No auth required.
3. **KRX Open API (openapi.krx.co.kr)** — EOD only, no intraday ranking. Ruled out per RESEARCH.

### Plan 02 Decision (Per Orchestrator Decision Rule)

**Decision rule from spike_scope_override:** "KRX blocked/auth-walled → Yahoo 5m cumulative fallback"

**Recommendation: Yahoo 5m cumulative fallback adapter for Plan 02.**

Rationale: KRX getJsonData.cmd requires authenticated session (not just OTP). Implementing login session management adds significant complexity and a KRX account credential dependency. Yahoo Finance 5m polling already exists in the codebase (YahooFinanceChartClient pattern) and can provide cumulative volume ranking without auth. This is the simpler, more reliable path for the live adapter.

**If orchestrator decides to pursue KRX_ID/KRX_PW path instead:** The OTP endpoint is accessible, and the pykrx login flow (POST to login URL → extract session cookie → use cookie on getJsonData.cmd) would be the implementation pattern.

### Deferred: Intraday Freshness Verification

Per spike_scope_override, "intraday freshness verification" (does ACC_TRDVOL update in near-real-time during 09:00–15:30 KST?) remains a **DEFERRED market-hours manual check**. This cannot be verified off-hours. If Plan 02 uses Yahoo fallback, this concern is moot for the KRX adapter. If KRX_ID/KRX_PW path is pursued, manual verification during market hours is required before production deployment.

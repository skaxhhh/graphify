### Gaps Summary

No gaps. All 5 success criteria are fully satisfied:

**SC1 — VolumeRankingProvider port + both impls:** The `VolumeRankingProvider` interface is a clean port contract. `DbVolumeRankingAdapter` provides the backtest impl using daily bar volume from `market_bars`. `YahooCumulativeVolumeAdapter` provides the live impl using 5m intraday cumulative volume from `market_bars_intraday`, with 1-min TTL cache. KRX MDC was auth-walled (HTTP 400 LOGOUT confirmed in Plan 01 spike) — Yahoo cumulative is the planned fallback, not a gap.

**SC2 — ETF/ETN/우선주 제외:** `V36__instrument_type.sql` adds the `instrument_type` column. Both adapters' JPQL queries filter `c.instrumentType = 'COMMON_STOCK'`. Integration tests (VolumeRankingProviderDbTest) confirm ETF and PREFERRED rows are excluded even when they have higher volume.

**SC3 — 매 틱 재선정 (not fixed at promote):** `VolumeRankRefresher.refreshIfVolumeTopN()` is called on every `LiveDataScheduler.collectLiveData()` tick (every 5 min during market hours), replacing `paper_live_symbols` via delete+insert before symbol ingestion begins.

**SC4 — 진입=top-N, 청산=보유 포함:** VolumeRankRefresher unions held-position symbols into the paper_live_symbols set, ensuring data is collected for them. LiveEvaluationService's `buildEntrySet()` gates new entry to top-N members only, while the exit branch (`posOpt.isPresent()`) evaluates unconditionally for all held symbols regardless of current top-N membership. Four targeted tests confirm the exact SC4 scenarios.

**SC5 — 문서화 + 일관성 방침:** `DESIGN.md ## [v1.5.0]` is the single-doc entry (per project documentation rules) containing the backtest↔live difference table, the explicit "차이를 수용·문서화, 백테스트 재정렬 안 함 (Deferred)" policy, and KRX deferred rationale. `VolumeRankingProvider.java` Javadoc contains the `@VolumeRankingSemantics` annotation documenting the semantic difference.

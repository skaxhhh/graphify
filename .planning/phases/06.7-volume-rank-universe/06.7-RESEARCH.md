## Summary

Phase 6.7는 세 가지 독립 문제의 결합이다: (1) KRX에서 "지금 시점 횡단면 거래량 순위" 데이터를 가져오는 것, (2) ETF/ETN/우선주를 구분할 메타데이터 컬럼 추가, (3) 라이브 평가 루프에서 종목을 매 틱 재선정하되 보유 포지션 종목은 이탈 후에도 청산 평가를 계속하는 로직 설계.

**가장 중요한 발견 — KRX MDC intraday 가용성:** KRX data.krx.co.kr의 MDCSTAT01501(전종목시세) 엔드포인트는 `getJsonData.cmd` POST로 당일(trdDd=오늘) 데이터를 조회할 수 있으며, 장 중에도 현재가·누적거래량이 실시간에 가깝게 갱신된다는 커뮤니티 사례가 있다. 그러나 2024년 이후 KRX는 인증(KRX_ID/KRX_PW 또는 Open API 키)을 요구하는 방향으로 전환했으며, 무인증 스크레이핑은 401 또는 빈 응답을 반환할 수 있다. 따라서 **계획 단계에서 반드시 실제 호출 테스트로 접근 가능 여부를 확인하고**, 실패 시 fallback 경로(Yahoo Finance per-symbol 5m intraday 누적 집계)를 플러그인 방식으로 준비해야 한다.

**Primary recommendation:** VolumeRankingProvider 포트를 `KrxMdcVolumeAdapter`(라이브)와 `DbVolumeAdapter`(백테스트)로 분리 구현하고, KRX 호출 실패 시 Yahoo 5m 누적 집계로 자동 전환하는 fallback 체인을 내장한다. 종목 분류는 `companies.instrument_type` VARCHAR 컬럼(COMMON_STOCK/ETF/ETN/PREFERRED/SPAC/ETC) 신규 마이그레이션으로 해결한다.

## Architecture Patterns

### VolumeRankingProvider 포트 설계

기존 `MarketDataPort` 패턴을 그대로 따른다. 소스: `backend/src/main/java/com/graphify/trading/engine/MarketDataPort.java`.

```
com.graphify.market/
├── volume/
│   ├── VolumeRankingProvider.java        # 포트 인터페이스
│   ├── DbVolumeRankingAdapter.java       # 백테스트 impl (findTopVolumeSymbolsOnDate)
│   └── KrxMdcVolumeAdapter.java          # 라이브 impl (KRX MDC HTTP + TTL 캐시)
```

```java
// VolumeRankingProvider.java
public interface VolumeRankingProvider {
    /**
     * 거래량 상위 N 티커 반환.
     * 백테스트: date = 해당 거래일, 완결된 일봉 기준.
     * 라이브:   date = 오늘, 당일 누적 거래량 기준.
     */
    List<String> topVolume(String market, LocalDate date, int topN, boolean excludeEtf);
}
```

### KRX MDC 호출 패턴

**엔드포인트 (HIGH confidence — pykrx 소스 확인):**

```
POST http://data.krx.co.kr/comm/bldAttendant/getJsonData.cmd
Content-Type: application/x-www-form-urlencoded
Referer: http://data.krx.co.kr/contents/MDC/MDI/mdiLoader/index.cmd?menuId=MDC0201
User-Agent: Mozilla/5.0 (compatible; graphify/0.1)

bld=dbms%2FMDC%2FSTAT%2Fstandard%2FMDCSTAT01501
&mktId=STK
&trdDd=YYYYMMDD
&share=1
&money=1
&csvxls_isNo=true
```

**파라미터 설명:**
- `bld`: `dbms/MDC/STAT/standard/MDCSTAT01501` — 전종목시세 (pykrx `전종목시세` 클래스에서 확인)
- `mktId`: `STK` = KOSPI, `KSQ` = KOSDAQ, `ALL` = 전체
- `trdDd`: 당일 날짜 YYYYMMDD
- `csvxls_isNo=true`: Excel 다운로드 없이 JSON 반환을 유도

**JSON 응답 구조 (MEDIUM confidence — pykrx 분석 + 커뮤니티):**
```json
{
  "output": [
    {
      "ISU_SRT_CD":  "005930",
      "ISU_ABBRV":   "삼성전자",
      "MKT_NM":      "KOSPI",
      "SECT_TP_NM":  "보통주",
      "ACC_TRDVOL":  "12345678",
      "ACC_TRDVAL":  "987654321000",
      "CMPPREVDD_PRC": "500",
      "FLUC_RT":     "0.71"
    }
  ]
}
```

**핵심 필드:**
- `ISU_SRT_CD`: 6자리 종목코드 (티커)
- `ISU_ABBRV`: 종목약명
- `SECT_TP_NM`: 종목구분 (`보통주`, `ETF`, `ETN`, `우선주`, `SPAC` 등)
- `ACC_TRDVOL`: 누적거래량 (당일 누적, 장 중 갱신)
- `MKT_NM`: 시장 (`KOSPI`, `KOSDAQ`)

**SECT_TP_NM 필터링으로 보통주만 선별 가능:** `"보통주"` 값만 포함하고 나머지 제외. 이것이 별도 ETF 마스터 없이 분류를 처리하는 가장 깔끔한 경로.

### 인증 상황 (CRITICAL — MEDIUM confidence, 반드시 Wave 0에서 실제 검증 필요)

2024년부터 KRX는 data.krx.co.kr 직접 호출에 로그인을 요구하는 방향으로 전환했다:
- 무인증 호출: 401 또는 빈 `output` 배열 반환 가능
- pykrx 최신(1.2.8): `KRX_ID`, `KRX_PW` 환경변수로 로그인 세션 쿠키를 획득한 뒤 getJsonData.cmd를 호출
- KRX Open API(openapi.krx.co.kr): API 키 신청 필요, 데이터는 주로 EOD(익일 오전 제공). 인트라데이 랭킹 엔드포인트 없음 (확인됨).

**계획 설계 함의 (CRITICAL):**

| 시나리오 | 대응 |
|--------|------|
| getJsonData.cmd 무인증 → 당일 데이터 정상 반환 | KrxMdcVolumeAdapter 그대로 사용, KRX_ID/PW 불필요 |
| getJsonData.cmd → 401 또는 빈 응답 | 세션 쿠키 로그인 경로 추가 (KRX_ID/PW env var) |
| 완전 차단 | Fallback: Yahoo 5m intraday 수집 결과를 symbol별 누적 합산으로 랭킹 계산 |

**Wave 0 필수 수동 검증:** `curl -X POST http://data.krx.co.kr/comm/bldAttendant/getJsonData.cmd -d "bld=dbms/MDC/STAT/standard/MDCSTAT01501&mktId=STK&trdDd=$(date +%Y%m%d)&csvxls_isNo=true" -H "Referer: http://data.krx.co.kr/" -H "User-Agent: Mozilla/5.0"` 응답 확인. → 결과에 따라 계획 Task 1 구현 전략 분기.

### KRX MDC 캐시 설계 (Manual TTL — Spring @Cacheable 미사용)

프로젝트에 Spring Cache 인프라(Caffeine Bean 등)가 없다. `IntradayBarCacheService` 패턴처럼 단순 구현 사용:

```java
// KrxMdcVolumeAdapter.java 내부
private volatile List<String> cachedRanking = List.of();
private volatile Instant cacheTs = Instant.EPOCH;
private static final Duration CACHE_TTL = Duration.ofMinutes(1);

public List<String> topVolume(String market, LocalDate date, int topN, boolean excludeEtf) {
    if (Instant.now().isBefore(cacheTs.plus(CACHE_TTL)) && !cachedRanking.isEmpty()) {
        return cachedRanking.subList(0, Math.min(topN, cachedRanking.size()));
    }
    List<String> fresh = fetchFromKrx(market, date, excludeEtf);
    if (!fresh.isEmpty()) {
        cachedRanking = fresh;
        cacheTs = Instant.now();
    }
    return cachedRanking.subList(0, Math.min(topN, cachedRanking.size()));
}
```

### 라이브 재선정 + 보유 포지션 청산 유지 설계

**현재 문제:**
- `PaperLifecycleService.resolveSymbols()` (line 176–195): `start()` 호출 시 `symbolsByMarket(market)`으로 전체 KOSPI 종목을 `paper_live_symbols`에 스냅샷 고정.
- `LiveEvaluationService.resolveSymbols()` (line 273–284): `paper_live_symbols` 테이블에서 읽어 종목 목록을 사용 — 고정 스냅샷.
- `LiveDataScheduler.collectLiveData()` (line 78): `symbolService.activeSymbolsUnion()` → `paper_live_symbols` 기반으로 수집 대상 결정.

**목표 설계:**

```
매 5분 틱:
  entrySymbols = VolumeRankingProvider.topVolume(market, today, topN, true)  // KRX 최신 top-N
  holdSymbols  = PaperPositionRepository.findByAccountId(accountId)
                   .map(PaperPosition::getSymbol).collect(Set)
  collectSymbols = entrySymbols ∪ holdSymbols  // 수집 대상 (ingest 5m bar)
  evalEntryFor  = entrySymbols   // 진입 신호 체크
  evalExitFor   = holdSymbols    // 청산 신호 체크 (top-N 이탈해도 포함)
```

**구현 접근법:**

`paper_live_symbols` 테이블의 역할이 바뀐다: `volume_top_n` 룰에서는 더 이상 고정 스냅샷이 아니라, 매 틱에 동적으로 계산되어야 한다. 두 가지 구현 경로:

**경로 A (추천 — 최소 변경):** `paper_live_symbols`를 그대로 유지하되, 매 틱 `LiveDataScheduler`가 수집 전에 volume_top_n 룰의 paper_live_symbols를 갱신(delete+insert)한다. 이후 기존 `activeSymbolsUnion()` → 수집 → `evaluateTick()` 흐름은 불변.

```java
// LiveDataScheduler.collectLiveData() 내부, 수집 전에 추가
refreshVolumeRankSymbols();  // volume_top_n 룰의 paper_live_symbols를 재선정
Set<String> symbols = symbolService.activeSymbolsUnion();  // 갱신된 값 포함
```

`refreshVolumeRankSymbols()`는: 각 RUNNING volume_top_n 룰에 대해 `topVolume(market, today, topN, true)` 호출 → 보유 포지션 종목 추가 → `paperLiveSymbolService.assignSymbols(ruleId, mergedSet)`.

**진입 게이팅:** `LiveEvaluationService.evaluateSymbol()` 내부에서, symbol이 현재 top-N에 있는지 추가로 체크하여 `positionOpt.isEmpty()`이면 진입 스킵 (top-N 이탈 종목은 진입 차단).

**경로 B:** `LiveEvaluationService`에 `VolumeRankingProvider` 주입, symbol 루프에서 직접 top-N 체크. — 더 복잡, 경로 A 선호.

### ETF/ETN 분류 전략

**SECT_TP_NM 기반 동적 필터 (추천):**
- KRX `getJsonData.cmd` 응답의 `SECT_TP_NM` 필드가 종목 구분을 포함한다.
- `SECT_TP_NM = "보통주"` 조건으로 필터링 → 별도 ETF 마스터 불필요.
- fallback으로 DB에 `instrument_type` 컬럼 추가 (마이그레이션 V36)도 병행하면, KRX 응답에 SECT_TP_NM이 없거나 Yahoo fallback 사용 시에도 필터 가능.

**DB 컬럼 방식 (보조):**
```sql
-- V36__instrument_type.sql
ALTER TABLE companies ADD COLUMN IF NOT EXISTS instrument_type VARCHAR(20) DEFAULT 'COMMON_STOCK';
-- ETF/ETN 종목 seed: KRX 전종목기본정보(MDCSTAT01901)로 일괄 분류
UPDATE companies SET instrument_type = 'ETF' WHERE ticker IN (...);
```

`Company.java`에 `instrument_type` 필드 추가 (`@Column(name = "instrument_type")`).

현재 `findTopVolumeSymbolsOnDate` JPQL 쿼리를 `instrument_type = 'COMMON_STOCK'` 조건으로 확장하면 백테스트도 일관성 확보.

### 백테스트 ↔ 라이브 일관성 방침 (Claude's Discretion — 추천)

| 구분 | 백테스트 | 라이브 |
|------|---------|-------|
| 기준 | 해당 거래일 완결 일봉 거래량 (market_bars) | 당일 누적 인트라데이 거래량 (KRX 장 중 갱신) |
| 범위 | in_kospi200=true 종목 (현재 약 110개) | 전체 KOSPI 보통주 (약 800개) |
| 의미 | 완결된 하루치 거래량으로 익일 전략 검증 | 당일 현재 시점까지의 누적으로 실시간 선정 |

**추천 방침: 차이를 수용·문서화, 백테스트 재정렬은 하지 않는다.**
- 이유: 백테스트 재정렬(인트라데이 누적 스냅샷 기준)은 V28 이전 데이터가 없고 비용이 크다. CONTEXT.md Deferred에 명시됨.
- 코드에 `@VolumeRankingSemantics` Javadoc 주석 등으로 명시.
- 백테스트 범위 확대(in_kospi200 → 전체 KOSPI 보통주)는 Phase 6.7 내에서 선택적으로 처리: `findTopVolumeSymbolsOnDate` 쿼리에서 `instrument_type = 'COMMON_STOCK'` 조건으로 KOSPI200 제약 해제 권장.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP POST with form encoding | 직접 URLEncoder 조립 | RestClient + `body(Map, APPLICATION_FORM_URLENCODED)` | Spring RestClient가 기본 제공, 프로젝트 표준 |

## Common Pitfalls

### Pitfall 1: KRX MDC 인증 미확인으로 런타임 실패
**What goes wrong:** 빌드·단위 테스트는 통과하지만 장 중 실제 KRX 호출이 401 또는 빈 배열을 반환해 top-N이 항상 비어 있음.
**Why it happens:** 2024년부터 KRX가 무인증 접근을 제한. 단위 테스트는 mock으로 통과.
**How to avoid:** Wave 0에서 실제 curl 호출로 응답 확인 필수. `enabled` 플래그 + fallback 체인 필수 구현.
**Warning signs:** `output` 배열이 비어 있거나 HTTP 401.

### Pitfall 2: 보유 포지션 종목을 top-N 재선정 시 수집 대상에서 누락
**What goes wrong:** 매 틱 paper_live_symbols를 새 top-N으로 교체하면, 이전 주기에 top-N이었다가 이탈한 종목의 5m 봉이 수집되지 않아 청산 신호 평가 불가.
**Why it happens:** `activeSymbolsUnion()`이 `paper_live_symbols`만 보기 때문.
**How to avoid:** `refreshVolumeRankSymbols()` 내에서 항상 보유 포지션 종목을 `paper_live_symbols`에 합산 후 저장.
**Warning signs:** 포지션이 있는 종목의 staleness 경고가 로그에 반복.

### Pitfall 3: volume_top_n이 아닌 룰(symbols/watchlist)에 재선정 로직 오적용
**What goes wrong:** `refreshVolumeRankSymbols()`가 모든 RUNNING 룰에 대해 KRX 호출 → symbols 타입 룰의 고정 종목 목록을 덮어씀.
**Why it happens:** 룰 타입 구분 없이 일괄 갱신.
**How to avoid:** `RuleDefinition.Universe.type` == `"volume_top_n"` 인 룰만 갱신 대상.

### Pitfall 4: 백테스트 findTopVolumeSymbolsOnDate가 KOSPI200(~110개)으로 한정되어 실제 거래량 상위 종목 누락
**What goes wrong:** KOSPI200에 포함 안 된 소형주·중형주가 거래량 1위여도 선정 안 됨.
**Why it happens:** `JOIN Company c ON c.inKospi200 = true` 조건.
**How to avoid:** `instrument_type` 컬럼 추가 후 JPQL에서 `c.instrumentType = 'COMMON_STOCK'`으로 조건 변경, inKospi200 제약 제거.
**Warning signs:** 백테스트 top-N 결과가 항상 대형주만 포함.

### Pitfall 5: KRX trdDd에 오늘 날짜 전달 시 장 개시 전 데이터 없음
**What goes wrong:** 09:00 이전 또는 장 중 초기에 당일 데이터가 아직 없어 빈 응답.
**Why it happens:** KRX 통계 시스템이 일정 시간 후 데이터 제공.
**How to avoid:** 빈 응답 시 전일 날짜로 fallback 재시도 로직 추가. 또는 9시 이전에는 평가 자체가 LiveDataScheduler의 09:00 guard로 차단됨 (이미 구현됨).

### Pitfall 6: `@DataJpaTest`에서 `instrument_type` 컬럼 마이그레이션 미적용
**What goes wrong:** H2 create-drop으로 Entity에 컬럼이 없어 테스트 실패.
**Why it happens:** Flyway 비활성화 (`spring.flyway.enabled=false`), JPA DDL이 Entity 기준으로 테이블 생성.
**How to avoid:** `Company.java`에 `instrument_type` 필드 추가 시 `@Column(columnDefinition)` 기본값 포함, Entity 변경이 H2 자동 생성에 반영됨.

## Code Examples

### KrxMdcVolumeAdapter — RestClient 호출 패턴

```java
// 소스: 프로젝트 YahooFinanceChartClient.java + pykrx MDCSTAT01501 분석
@Component
public class KrxMdcVolumeAdapter implements VolumeRankingProvider {

    private static final String KRX_JSON_URL =
        "http://data.krx.co.kr/comm/bldAttendant/getJsonData.cmd";
    private static final Duration CACHE_TTL = Duration.ofMinutes(1);

    private final RestClient restClient;
    private volatile List<String> cache = List.of();
    private volatile Instant cacheTs = Instant.EPOCH;

    public KrxMdcVolumeAdapter(RestClient.Builder builder) {
        this.restClient = builder
            .baseUrl(KRX_JSON_URL)
            .defaultHeader("Referer",
                "http://data.krx.co.kr/contents/MDC/MDI/mdiLoader/index.cmd")
            .defaultHeader("User-Agent", "Mozilla/5.0 (compatible; graphify/0.1)")
            .build();
    }

    @Override
    public List<String> topVolume(String market, LocalDate date, int topN, boolean excludeEtf) {
        if (Instant.now().isBefore(cacheTs.plus(CACHE_TTL)) && !cache.isEmpty()) {
            return cache.subList(0, Math.min(topN, cache.size()));
        }
        String trdDd = date.format(DateTimeFormatter.BASIC_ISO_DATE); // YYYYMMDD
        try {
            JsonNode root = restClient.post()
                .contentType(MediaType.APPLICATION_FORM_URLENCODED)
                .body("bld=dbms%2FMDC%2FSTAT%2Fstandard%2FMDCSTAT01501"
                    + "&mktId=STK&trdDd=" + trdDd + "&csvxls_isNo=true")
                .retrieve()
                .body(JsonNode.class);

            List<String> fresh = parseRanking(root, excludeEtf);
            if (!fresh.isEmpty()) { cache = fresh; cacheTs = Instant.now(); }
        } catch (Exception ex) {
            log.warn("KRX MDC volume ranking fetch failed: {}", ex.getMessage());
            // fallback: return stale cache or empty
        }
        return cache.subList(0, Math.min(topN, cache.size()));
    }

    private List<String> parseRanking(JsonNode root, boolean excludeEtf) {
        if (root == null || !root.has("output")) return List.of();
        List<String> result = new ArrayList<>();
        for (JsonNode item : root.path("output")) {
            String sectTp = item.path("SECT_TP_NM").asText("");
            if (excludeEtf && !"보통주".equals(sectTp)) continue;
            String ticker = item.path("ISU_SRT_CD").asText("");
            if (!ticker.isBlank()) result.add(ticker);
        }
        return result; // KRX returns already sorted by ACC_TRDVOL DESC
    }
}
```

## Open Questions

1. **KRX getJsonData.cmd 무인증 접근 가능 여부**
   - What we know: 2024년부터 인증 강화 방향. pykrx 1.2.8은 KRX_ID/PW 사용. 하지만 일부 엔드포인트는 여전히 Referer 헤더만으로 접근 가능하다는 커뮤니티 보고 있음.
   - What's unclear: 현재(2026-06) data.krx.co.kr MDCSTAT01501의 무인증 접근 가능 여부.
   - Recommendation: **Wave 0 Task 1에서 curl 테스트 필수 선행.** 차단 시 KRX_ID/PW env var 방식 구현. 완전 차단 시 Yahoo 5m 누적 fallback 활성화.

2. **KRX MDCSTAT01501이 장 중 실시간(또는 근실시간)으로 ACC_TRDVOL을 갱신하는가**
   - What we know: 전종목시세 화면은 장 중에 갱신됨. 정확한 갱신 주기 미확인.
   - What's unclear: 5분 지연인지, 1분 지연인지, 실시간인지.
   - Recommendation: Wave 0에서 장 중 동일 날짜로 2회 이상 호출하여 ACC_TRDVOL 변화 확인. 갱신 주기가 5분 이상이면 1분 TTL 캐시로 충분.

3. **instrument_type 컬럼 초기 시드 범위**
   - What we know: `companies` 테이블에 현재 약 6개 실제 데이터 (V30 SQL 기준으로 110개 티커 IN-list이지만 로컬 DB 실수입은 6개).
   - What's unclear: 전체 KOSPI 보통주 800개를 언제, 어떻게 추가할 것인가.
   - Recommendation: V36 마이그레이션에서 기존 ticker에 대한 `instrument_type = 'COMMON_STOCK'` 기본값 설정. 전체 KOSPI 종목 seed는 KRX MDCSTAT01901(전종목기본정보)로 별도 초기화 스크립트 작성. Phase 6.7 범위에서 최소화: 기존 companies 내 종목 분류만 처리.

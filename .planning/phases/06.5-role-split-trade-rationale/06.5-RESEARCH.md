## Summary

Phase 6.5는 세 가지 독립 레이어의 변경으로 구성된다.

**첫째, DB 상태 모델 재구성.** 현재 `trading_rules.status` 컬럼은 단일 문자열(`DRAFT`, `BACKTESTED`, `PAPER_LIVE`, `PAUSED`)로 config와 run 의미를 혼합한다. 연구 결과 신규 컬럼 두 개(`config_status`, `run_status`)를 추가하는 Flyway 마이그레이션이 가장 명확하고 안전하다. 기존 행의 마이그레이션 매핑(`PAPER_LIVE → ACTIVE/RUNNING`, `PAUSED → ACTIVE/STOPPED`, `DRAFT/BACKTESTED → DRAFT/STOPPED`)을 마이그레이션 SQL에서 한 번에 처리한다. 이렇게 하면 `RuleStatus.isLiveActive()` 단일 편집 지점을 `run_status = 'RUNNING'` 기준으로 깔끔하게 전환할 수 있다.

**둘째, 백엔드 근거 계측.** `RuleEvaluator`의 `entryTriggered`/`exitTriggered`가 현재 boolean만 반환한다. 새 리치 반환 타입 `EvalResult`(충족 조건식 리스트 + 조건별 실제 지표값 + 청산 reason enum)를 도입한다. `IntradayBacktestEngine`의 진입/청산 결정 지점(라인 125–133)과 `LiveEvaluationService`의 신호 결정 지점(라인 165–185)에서 `EvalResult`를 포착해 각각 `BacktestResult.TradeDto`(확장)와 `paper_signal_log.indicatorSnapshot` JSONB에 근거를 삽입한다. 백테스트는 인라인 확장(TradeDto에 rationale 필드 추가), 모의는 `paper_signal_log`에서 JOIN(`rule_id + symbol + ts` 기준)으로 조회한다.

**셋째, 프론트엔드 UI 분리 + 아코디언.** `PaperRulesPage`(전략 설정)는 config축 배지·토글만 노출하고, `TradingRulesPage`(전략 운영)는 ACTIVE 룰만 필터링해 run축 배지·시작/중지만 노출한다. 거래 테이블 양쪽에 `TradeRationaleRow` 확장 컴포넌트를 아코디언 패턴으로 삽입한다.

**Primary recommendation:** 신규 컬럼 마이그레이션(V35) + `EvalResult` 리치 반환 타입 + 아코디언 확장 행 컴포넌트 재사용 패턴으로 구현한다.

## Discretion Area Resolutions

### 1. 2축 상태 DB 표현 — 신규 컬럼 마이그레이션 권장

**결론: 신규 컬럼 2개 추가 (V35 마이그레이션)**

**근거:**
- 기존 `status` 컬럼은 `VARCHAR(16) NOT NULL DEFAULT 'DRAFT'`(V27 확인). 문자열 파생 매핑(예: `PAPER_LIVE` → config=ACTIVE/run=RUNNING)은 서비스 레이어에 변환 로직이 분산되고, `@Query` JPQL에서 양쪽 축을 동시에 쿼리할 때 CASE 식이 복잡해진다.
- 신규 컬럼 방식은 `WHERE run_status = 'RUNNING'`으로 LiveEvaluationService 게이트 단순화가 가능하고, `RuleStatus.isLiveActive()`를 단일 편집 지점으로 유지하면서 `run_status` 컬럼 체크로 교체할 수 있다.
- 기존 행 마이그레이션 매핑은 안전하게 SQL UPDATE로 처리 가능: `PAPER_LIVE → (ACTIVE, RUNNING)`, `PAUSED → (ACTIVE, STOPPED)`, `DRAFT/BACKTESTED → (DRAFT, STOPPED)`.
- `status` 컬럼은 호환성을 위해 일시 보존 후 Phase 7에서 제거(옵션) 또는 그대로 유지.

**마이그레이션 SQL 패턴:**
```sql
-- V35__two_axis_status.sql
ALTER TABLE trading_rules
    ADD COLUMN IF NOT EXISTS config_status VARCHAR(8) NOT NULL DEFAULT 'DRAFT',
    ADD COLUMN IF NOT EXISTS run_status    VARCHAR(8) NOT NULL DEFAULT 'STOPPED';

-- 기존 행 마이그레이션
UPDATE trading_rules SET config_status = 'ACTIVE', run_status = 'RUNNING'
    WHERE status = 'PAPER_LIVE';
UPDATE trading_rules SET config_status = 'ACTIVE', run_status = 'STOPPED'
    WHERE status = 'PAUSED';
UPDATE trading_rules SET config_status = 'DRAFT', run_status = 'STOPPED'
    WHERE status IN ('DRAFT', 'BACKTESTED');

CREATE INDEX IF NOT EXISTS idx_trading_rules_run_status ON trading_rules (run_status);
```

**TradingRule.java 확장:**
```java
@Column(name = "config_status", nullable = false)
private String configStatus = "DRAFT";  // DRAFT | ACTIVE

@Column(name = "run_status", nullable = false)
private String runStatus = "STOPPED";   // STOPPED | RUNNING
```

**RuleStatus.java 변경:**
```java
// 기존: PAPER_LIVE.equals(status)
// 신규: "RUNNING".equals(rule.getRunStatus())
public static boolean isLiveActive(TradingRule rule) {
    return "RUNNING".equals(rule.getRunStatus());
}
```

### 4. 백테스트 trade ↔ 근거 연결

**결론: `BacktestResult.TradeDto` 인라인 확장**

백테스트는 in-memory `PaperLedger`를 사용하므로 DB 조인이 없다. `IntradayBacktestEngine.run()` 루프의 진입/청산 결정 지점(라인 125–133)에서 `evalEntry`/`evalExit`를 호출해 `EvalResult`를 포착하고, `PaperLedger.buy()/sell()` 호출 직후 rationale을 TradeRecord에 연결한다.

**구체적 변경:**
1. `PaperLedger.TradeRecord`에 `String rationaleJson` 필드 추가
2. `PaperLedger.buy()/sell()`에 `rationaleJson` 파라미터 추가
3. `BacktestResult.TradeDto`에 `String rationaleJson` 필드 추가
4. `IntradayBacktestEngine.buildResult()`에서 TradeRecord → TradeDto 매핑 시 rationaleJson 전달
5. 프론트 `BacktestTrade` 타입에 `rationaleJson: string | null` 추가

### 5. 모의 trade ↔ 근거 연결

**결론: `paper_signal_log` JOIN 방식 (비정규화 없음)**

`paper_trades`는 이미 `rule_id`, `symbol`, `traded_at` 컬럼을 가진다(PaperTrade.java 확인). `paper_signal_log`는 `rule_id`, `symbol`, `ts`, `indicator_snapshot`을 가진다(PaperSignalLog.java + V32 확인).

JOIN 키: `rule_id = rule_id AND symbol = symbol AND ts = traded_at` (PaperExecutor가 동일 `ts`를 signal log와 trade 모두에 기록).

### 6. PaperLifecycleService 2축 재구성

**결론: 기존 promote/pause/resume을 activate/start/stop으로 의미 재구성**

기존 메서드와 새 의미의 매핑:

| 기존 | 신규 | 변경 내용 |
|------|------|----------|
| `promote(DRAFT→PAPER_LIVE)` | `activate(DRAFT→ACTIVE)` | 백테스트 gate 제거, config_status만 변경 |
| `pause(PAPER_LIVE→PAUSED)` | `stop(ACTIVE/RUNNING→ACTIVE/STOPPED)` | run_status만 STOPPED로 변경 |
| `resume(PAUSED→PAPER_LIVE)` | `start(ACTIVE/STOPPED→ACTIVE/RUNNING)` | run_status만 RUNNING으로 변경, paperLiveSymbols 재할당 |
| `copy()` | `copy()` | 변경 없음 |

신규 엔드포인트 매핑 (`PaperLifecycleController`):
```
POST /api/v1/trading/paper/rules/{id}/activate  ← DRAFT → ACTIVE
POST /api/v1/trading/paper/rules/{id}/deactivate ← ACTIVE/STOPPED → DRAFT (RUNNING 시 차단)
POST /api/v1/trading/paper/rules/{id}/start     ← ACTIVE/STOPPED → ACTIVE/RUNNING
POST /api/v1/trading/paper/rules/{id}/stop      ← ACTIVE/RUNNING → ACTIVE/STOPPED
POST /api/v1/trading/paper/rules/{id}/copy      ← 기존 유지
```

기존 `/promote`, `/pause`, `/resume` 엔드포인트는 프론트엔드 `paperApi.ts`에서 이미 참조하므로 신규 엔드포인트 추가 후 프론트 API 함수도 교체한다.

## Common Pitfalls

### Pitfall 1: `status` 컬럼 완전 제거 시 기존 JPA 쿼리 파손
**What goes wrong:** `PaperRuleService.STATUSES = Set.of("DRAFT", "ACTIVE", "PAUSED")`가 이미 존재하는데, 기존 `status` 컬럼을 제거하면 `findByUserIdAndModeOrderByUpdatedAtDesc()` 등 기존 JPA 메서드가 그대로 작동하더라도 `PaperLifecycleService.promote()`의 `"DRAFT".equals(rule.getStatus())` 가드가 깨진다.
**Prevention:** 신규 컬럼 추가 후 `status` 컬럼은 당분간 유지. `PaperLifecycleService`의 상태 게이트를 `configStatus`/`runStatus` 기준으로 교체하되 `getStatus()` 호출을 모두 찾아 교체.
**Warning signs:** 컴파일 오류보다는 런타임에 잘못된 상태 전이 허용으로 나타남. 단위 테스트로 전이 가드 검증 필수.

### Pitfall 2: `paper_signal_log` ↔ `paper_trades` ts 매칭 정확도
**What goes wrong:** `PaperExecutor.execute()`에서 `ts` 파라미터가 `LiveEvaluationService.evaluateSymbol()`의 `tickTime`과 동일하게 전달되지만, signal log는 HOLD 신호도 기록한다(라인 55). 매수 체결 시에는 BUY 신호 로그, 매도 체결 시에는 SELL 신호 로그를 찾아야 하므로 `signal = trade.getSide()` 조건도 JOIN에 포함해야 한다.
**Prevention:** JOIN 쿼리: `rule_id + symbol + ts + signal(=side)` 조합.
**Warning signs:** 동일 ts에 BUY+SELL 로그가 모두 존재하는 경우(현실적으로 드물지만 방어적으로 처리).

### Pitfall 3: RuleEvaluator `crossAbove`/`crossBelow` 조건의 리치 반환
**What goes wrong:** `crossAbove`/`crossBelow`는 `i`와 `i-1` 두 인덱스의 값이 필요(RuleEvaluator.java:67–80). `ConditionResult`에 어떤 값을 기록할지 불명확.
**Prevention:** 크로스 조건은 `leftValue = lNow`, `rightValue = rNow` (현재 인덱스 값)로 기록. `expr`에 크로스 방향 포함: `"EMA(9) crossAbove SMA(20)"`.

### Pitfall 4: 백테스트 `TradeDto`의 `datetime`이 date-only
**What goes wrong:** 현재 `IntradayBacktestEngine.buildResult()`에서 `t.date().atStartOfDay()`로 매핑(라인 241). 즉 시분초 정보가 없다. 근거 연결은 인덱스 기반(같은 TradeRecord 객체에서 rationale 포착)이므로 ts 매칭 문제 없다.
**Prevention:** 백테스트 TradeRecord에 `LocalDate date` 대신 `LocalDateTime datetime`을 추가하거나, rationale을 같은 객체에 인라인으로 들고 다니면 된다(권장: 인라인).

### Pitfall 5: `PaperRuleService.STATUSES` 화이트리스트 부재
**What goes wrong:** `PaperRuleService.normalizeStatus()`의 `STATUSES = Set.of("DRAFT", "ACTIVE", "PAUSED")`에 새 상태가 없다. `configStatus` 기반으로 변경하면 `ACTIVE`를 write 경로에서 허용해야 한다.
**Prevention:** CRUD 생성/수정 시 `config_status`만 설정 가능하게 제한. `run_status`는 lifecycle 엔드포인트에서만 변경.

### Pitfall 6: `RuleResponse` DTO에 새 필드 추가 누락
**What goes wrong:** `RuleResponse` record가 `(id, name, mode, status, backtested, definition, promotedFrom, createdAt, updatedAt)` 형태(PaperLifecycleService.toResponse() 확인). `configStatus`와 `runStatus`를 추가하지 않으면 프론트에서 두 필드가 null로 수신.
**Prevention:** `RuleResponse` record에 `configStatus`, `runStatus` 필드 추가 필수.

### Pitfall 7: MON-05 PRICE_LIMIT_PENDING
**What goes wrong:** 상한가/하한가 종목은 `FillSimulator.fillPrice()`가 참조가격 그대로를 반환하지만, 실제 체결 불가능한 상태다. 현재 `PaperExecutor`는 이를 감지하지 않는다.
**Prevention:** `PRICE_LIMIT_PENDING` 처리는 signal log의 `signal` 필드에 새 값으로 기록하거나, executed=false + reason 필드로 기록한다. 현재 paper_signal_log 스키마는 `signal VARCHAR(8)`이므로 `PENDING` 값 길이가 맞는다(7자). 체결 실행 전 가격 변동률 체크(`|price - prevClose| / prevClose >= 0.295`)로 감지.

```typescript
// RSI(14)=28.3 < 30 ✓ 형식
export function TradeRationaleRow({ rationale }: { rationale: TradeRationale }) {
  if (!rationale) return null;
  const exitLabel = rationale.exitReason === "TAKE_PROFIT"
    ? `익절 (+${rationale.exitPct?.toFixed(1)}%)`
    : rationale.exitReason === "STOP_LOSS"
    ? `손절 (${rationale.exitPct?.toFixed(1)}%)`
    : rationale.exitReason === "INDICATOR"
    ? "지표 조건"
    : null;

  return (
    <div className="space-y-1 text-xs">
      {exitLabel && (
        <div className={rationale.exitReason === "TAKE_PROFIT" ? "text-emerald-400" : "text-red-400"}>
          청산 사유: {exitLabel}
        </div>
      )}
      {rationale.conditions.map((c, i) => (
        <div key={i} className="flex items-center gap-2">
          <span className={c.passed ? "text-emerald-400" : "text-red-400"}>
            {c.passed ? "✓" : "✗"}
          </span>
          <span className="text-gray-300">
            {c.leftLabel}={c.leftValue.toFixed(2)} {c.op} {c.rightValue.toFixed(2)}
          </span>
        </div>
      ))}
    </div>
  );
}
```

## Open Questions

1. **`status` 컬럼 보존 기간**
   - What we know: 기존 `RuleResponse`가 `status` 필드를 반환하고, 프론트 `TradingRule.status` 타입이 이를 참조한다.
   - What's unclear: `status` 컬럼을 즉시 제거할지, 일정 기간 동기화 유지할지.
   - Recommendation: Phase 6.5에서는 `status` 컬럼을 유지하되 write 경로에서는 `config_status`/`run_status`로 전환. `RuleResponse`에 두 새 필드 추가 후 프론트는 `configStatus`/`runStatus`를 사용. 기존 `status` 필드는 deprecated 처리.

2. **마지막 신호 시각 데이터 소스**
   - What we know: 전략 운영 화면에 "마지막 신호 시각" 컬럼 필요. `paper_signal_log`에 `ts` 컬럼 존재.
   - What's unclear: per-rule max(ts) 쿼리를 별도 API로 노출할지, `/paper/rules-lifecycle` API가 JOIN으로 포함할지.
   - Recommendation: `PaperLifecycleController`에 GET `/paper/rules/lifecycle` 엔드포인트를 신규 추가해 ACTIVE 룰 + max(signal_log.ts) 집계를 한 번에 반환하는 DTO를 제공.

3. **MON-05 가격 변동률 임계값 소스**
   - What we know: PRICE_LIMIT_PENDING 감지를 위해 전일 종가 참조 필요.
   - What's unclear: 전일 종가를 `market_bars`에서 조회할지(일봉), 최근 인트라데이 bars에서 파생할지.
   - Recommendation: `market_bars`(일봉)에서 전일 종가 조회. KRX 상하한가 임계 ±29.5% 하드코딩.

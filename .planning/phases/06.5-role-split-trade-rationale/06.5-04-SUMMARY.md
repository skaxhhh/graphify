# Phase 6.5 Plan 04: Paper Live Rationale Instrumentation + MON-05 Summary

**One-liner:** LiveEvaluationService uses evalEntry/evalExit to merge rationale block into indicatorSnapshot; PaperHistoryService JOINs paper_signal_log for per-trade rationale; PaperExecutor detects KRX ±29.5% price limits as PRICE_LIMIT_PENDING before execution.

## What Was Built

### Task 1: LiveEvaluationService EvalResult + rationale merge

`evaluateSymbol()` now calls `ruleEvaluator.evalEntry()` / `ruleEvaluator.evalExit()` (EvalResult rich return from Plan 02) instead of the boolean `entryTriggered`/`exitTriggered`.

When BUY or SELL signal fires, `mergeRationale(snapshotJson, evalResult, side)` merges a `rationale` block into the existing `{"price", "rsi14", "sma20"}` snapshot JSON:

```json
{
  "price": 70000.0,
  "rsi14": 28.3,
  "sma20": 68000.0,
  "rationale": {
    "side": "BUY",
    "exitReason": null,
    "exitPct": null,
    "conditions": [
      {"expr": "RSI(14) < 30", "leftLabel": "RSI(14)", "leftValue": 28.3,
       "op": "<", "rightLabel": "30.0", "rightValue": 30.0, "passed": true}
    ]
  }
}
```

HOLD signals pass the plain snapshot (no rationale — saves storage).

The merged snapshot flows into `PaperExecutor.execute()` → `saveSignalLog()` → `paper_signal_log.indicator_snapshot` JSONB.

### Task 2: PaperHistoryService JOIN + PaperTradeHistoryItem.rationaleJson

`PaperTradeHistoryItem` record gains a `String rationaleJson` field (9th component).

`PaperHistoryService.getHistory()` injects `PaperSignalLogRepository` and, for each trade, looks up the matching signal log via:

```
findFirstByRuleIdAndSymbolAndTsAndSignal(ruleId, symbol, tradedAt, side)
```

The `signal=side` condition (Pitfall 2) defends against same-ts BUY+SELL log collision. `rationaleJson` is the `indicator_snapshot` string from the matched log (null if no log found or ruleId is null).

**For Plan 05 frontend:** `PaperTradeHistoryItem.rationaleJson` is the field to consume. It contains the full merged JSON including the `rationale` block — parse with `JSON.parse(item.rationaleJson)?.rationale` to get the `TradeRationale` object.

### Task 3: MON-05 PRICE_LIMIT_PENDING

`PaperExecutor` receives `MarketBarRepository` and exposes:

```java
boolean isPriceLimitPending(String symbol, double price, Instant ts)
```

Logic: look up `market_bars` for `(symbol, prevDate)` where `prevDate = tickDate(KST) - 1 day`. If `|price - prevClose| / prevClose >= 0.295` → `true`.

When true, execution is skipped and `signal='PENDING', executed=false` is logged. `TradeResult.skipped("BUY"/"SELL", symbol, "PRICE_LIMIT_PENDING")` is returned.

This check runs before the account lookup, so no account/position state is touched for pending-limit trades.

## Decisions Made

1. **mergeRationale schema**: `rationale` block nested inside existing `indicatorSnapshot` root — same schema as Plan 03 backtest `buildRationale()`. Frontend can share the `TradeRationale` parser.

2. **JOIN key**: `rule_id + symbol + ts + signal(=side)` — `signal=side` prevents BUY trade from picking up a SELL log at same timestamp.

3. **prevDate = tickDate - 1 calendar day**: Simplest approximation for "전일". Weekends/holidays may return empty (no trading bar) → falls back to `false` (allow fill). Acceptable for PAPER mode.

4. **PENDING in VARCHAR(8)**: "PENDING" is 7 characters, fits the existing column constraint without migration.

5. **lenient() default stub**: `HOLD` path returns before reaching `isPriceLimitPending`, so the default `marketBarRepo` stub must be lenient to avoid Mockito's UnnecessaryStubbingException.

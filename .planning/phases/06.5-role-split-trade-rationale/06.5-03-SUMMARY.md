**진입/청산 결정 지점 교체 (RESEARCH E2 패턴):**
```java
if (ledger.holds(symbol)) {
    EvalResult exitResult = ruleEvaluator.evalExit(def.exit(), closes, volumes, i, entryPrice);
    if (exitResult.triggered()) {
        String rationaleJson = buildRationale(exitResult, "SELL");
        ledger.sell(date, symbol, closes[i], rationaleJson);
        lastExitIndex.put(symbol, gIdx);
    }
} else {
    EvalResult entryResult = ruleEvaluator.evalEntry(def.entry(), closes, volumes, i);
    if (entryResult.triggered()) {
        String rationaleJson = buildRationale(entryResult, "BUY");
        double qty = sizeQty(def.sizing(), ledger.cash(), closes[i]);
        ledger.buy(date, symbol, qty, closes[i], rationaleJson);
    }
}
```

**buildRationale 헬퍼** (RESEARCH Discretion 2 스키마 정확히 일치):
```java
private String buildRationale(EvalResult r, String side) {
    Map<String, Object> map = new LinkedHashMap<>();
    map.put("side", side);
    map.put("exitReason", r.exitReason() != null ? r.exitReason().name() : null);
    map.put("exitPct", r.exitPct());
    // conditions 배열 ...
    map.put("conditions", conditions);
    try { return objectMapper.writeValueAsString(map); }
    catch (JsonProcessingException e) { log.warn(...); return null; }
}
```

**buildResult() 매핑** — `t.rationaleJson()` → `TradeDto` 8번째 인수로 전달.

### Rationale JSON 스키마 (Plan 05 프론트 파서 참조)

BUY 진입:
```json
{
  "side": "BUY",
  "exitReason": null,
  "exitPct": null,
  "conditions": [
    {
      "expr": "RSI(14) < 30",
      "leftLabel": "RSI(14)",
      "leftValue": 28.57,
      "op": "<",
      "rightLabel": "30.0",
      "rightValue": 30.0,
      "passed": true
    }
  ]
}
```

SELL 청산 (TP/SL):
```json
{
  "side": "SELL",
  "exitReason": "STOP_LOSS",
  "exitPct": -3.5,
  "conditions": []
}
```

SELL 청산 (INDICATOR):
```json
{
  "side": "SELL",
  "exitReason": "INDICATOR",
  "exitPct": null,
  "conditions": [
    { "expr": "PRICE < 980.0", "leftLabel": "PRICE", "leftValue": 975.0,
      "op": "<", "rightLabel": "980.0", "rightValue": 980.0, "passed": true }
  ]
}
```

**필드명 정확도:** `side`, `exitReason`, `exitPct`, `conditions[].{expr, leftLabel, leftValue, op, rightLabel, rightValue, passed}` — Plan 05 프론트 파서가 이 정확한 필드명을 사용해야 한다.

## Decisions Made

1. **PaperLedger 오버로드 위임**: `buy(4-arg)` → `buy(5-arg, null)` 위임. 기존 호출자(IntradayBacktestEngine 이외의 경로) 하위 호환.

2. **ObjectMapper 생성자 주입**: Spring 싱글턴 `ObjectMapper` 빈 재사용 (기존 다른 컴포넌트 컨벤션과 동일).

3. **buildRationale에서 LinkedHashMap 사용**: JSON 필드 순서 보장 (side → exitReason → exitPct → conditions 순).

4. **JsonProcessingException 처리**: `log.warn` + `null` 반환 — rationale 실패가 백테스트 자체를 중단하지 않아야 하므로.

5. **백테스트 = 인라인 운반 방식 (RESEARCH Discretion 4)**: DB 조인 없음. 동일 TradeRecord 객체에 rationaleJson 인라인 보유. datetime이 date-only(`atStartOfDay()`)여도 ts 매칭 불필요.

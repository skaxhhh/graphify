## Summary

Phase 6.9는 순수 프론트 작업이 아니다. **백엔드에 신규 도메인 엔티티(PaperRun)를 도입하고, 기존 paper_trades에 run_id를 태깅하며, run-scoped REST API를 신설한 다음 프론트 IA를 재구성한다.**

현재 코드베이스 상태: `paper_trades`에 `rule_id`는 이미 존재하지만 `run_id`는 없다. `paper_positions`와 `paper_equity_snapshots`에는 rule_id/run_id 모두 없다. start/stop은 `trading_rules.run_status`를 토글할 뿐 이력을 남기지 않는다. 가장 최신 Flyway 마이그레이션은 `V36__instrument_type.sql`이므로 이 phase의 마이그레이션은 **V37**이다.

설계 핵심: (1) `PaperLifecycleService.start()`에서 `paper_runs` 행을 생성하고, `stop()`에서 종료 처리한다. (2) `PaperExecutor.executeBuy/Sell()`에서 현재 RUNNING run을 조회해 `paper_trades.run_id`를 스탬핑한다. (3) "이 run의 포지션"은 별도 컬럼 없이 trades에서 파생한다(`run_id=X인 BUY 중 대응 SELL 없는 종목`). (4) 기여분 집계(실현손익·거래수·미실현손익)는 trades를 run_id로 필터하고, 총자산/현금은 account 전체 값을 사용한다.

**Primary recommendation:** Backend Wave(V37 migration + PaperRun entity + start/stop run lifecycle + PaperExecutor run_id stamping + 3 new run-scoped controllers) → Frontend Wave(새 routes/pages + TradingLayout nav 재편 + TradingRulesPage 상단 위젯).

## Run Data Model Proposal

### V37 Migration (next free: V36 is latest)

파일: `backend/src/main/resources/db/migration/V37__paper_runs.sql`

```sql
-- V37: paper_runs 엔티티 도입 + paper_trades에 run_id 태깅
-- run = 룰 start→stop 1회 (D6). 재실행 = 새 행(회차).

CREATE TABLE IF NOT EXISTS paper_runs (
    id                BIGSERIAL    PRIMARY KEY,
    rule_id           BIGINT       NOT NULL,
    user_id           BIGINT       NOT NULL,
    started_at        TIMESTAMPTZ  NOT NULL DEFAULT now(),
    ended_at          TIMESTAMPTZ,                        -- NULL = 진행중
    status            VARCHAR(8)   NOT NULL DEFAULT 'RUNNING',  -- RUNNING | STOPPED
    universe_snapshot TEXT,                               -- JSON array: 시작 시점 종목 목록
    CONSTRAINT chk_paper_run_status CHECK (status IN ('RUNNING','STOPPED'))
);

CREATE INDEX IF NOT EXISTS idx_paper_runs_rule  ON paper_runs (rule_id);
CREATE INDEX IF NOT EXISTS idx_paper_runs_user  ON paper_runs (user_id, started_at DESC);
CREATE INDEX IF NOT EXISTS idx_paper_runs_status ON paper_runs (status);

-- paper_trades에 run_id 추가 (rule_id는 이미 존재)
ALTER TABLE paper_trades
    ADD COLUMN IF NOT EXISTS run_id BIGINT;

CREATE INDEX IF NOT EXISTS idx_paper_trades_run ON paper_trades (run_id);
-- 기존 trades는 run_id = NULL (go-forward only — 하단 Backfill 전략 참조)
```

이 phase에서 `paper_positions`와 `paper_equity_snapshots`는 **변경하지 않는다**. 포지션은 계좌 전체 공유(D5). 스냅샷은 ts 범위로 run 기간을 필터한다.

### JPA Entity: PaperRun

파일: `backend/src/main/java/com/graphify/trading/paper/PaperRun.java`

```java
@Entity
@Table(name = "paper_runs")
public class PaperRun {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "rule_id", nullable = false)
    private Long ruleId;

    @Column(name = "user_id", nullable = false)
    private Long userId;

    @Column(name = "started_at", nullable = false)
    private Instant startedAt;

    @Column(name = "ended_at")
    private Instant endedAt;

    @Column(nullable = false, length = 8)
    private String status;  // RUNNING | STOPPED

    @Column(name = "universe_snapshot")
    private String universeSnapshot;  // JSON string, e.g. '["005930","000660"]'

    protected PaperRun() {}

    public PaperRun(Long ruleId, Long userId, Instant startedAt, String universeSnapshot) {
        this.ruleId = ruleId;
        this.userId = userId;
        this.startedAt = startedAt;
        this.status = "RUNNING";
        this.universeSnapshot = universeSnapshot;
    }

    public void stop(Instant endedAt) {
        this.endedAt = endedAt;
        this.status = "STOPPED";
    }

    // getters omitted for brevity
}
```

### Backfill Strategy

**Go-forward only.** 기존 `paper_trades`(run_id = NULL)는 run-scoped 조회에서 제외된다.

단, "전략 전체 통합" 모드(D7 mode 2)는 `rule_id + 기간` 필터를 사용하므로 run_id=NULL인 구형 거래도 포함된다. 이 방식으로 구형 데이터를 잃지 않는다.

현재 `run_status='RUNNING'`인 룰이 있다면(Phase 6.9 배포 시점): Phase 6.9 start/stop이 새 run을 만들기 전까지 해당 룰은 orphaned RUNNING 상태다. 이를 위해 V37 마이그레이션에서 다음을 추가한다:

```sql
-- 현재 RUNNING 중인 룰에 대해 synthetic 최초 run 생성 (backfill 1건)
INSERT INTO paper_runs (rule_id, user_id, started_at, status, universe_snapshot)
SELECT tr.id, tr.user_id, tr.updated_at, 'RUNNING', NULL
FROM trading_rules tr
WHERE tr.run_status = 'RUNNING';
```

이 백필은 간단하고 안전하다(existing RUNNING rule만 대상, universe_snapshot=NULL 허용).

## Contribution Aggregation Spec

D5(단일 계좌 공유) 전제에서 "run의 기여분"을 어떻게 계산하는가.

### 실행 이력 리스트 컬럼 계산

| 컬럼 | 계산 방법 | 스코프 |
|------|----------|--------|
| 전략명 | `trading_rules.name` WHERE rule_id = run.ruleId | — |
| 상태 | `paper_runs.status` (RUNNING/STOPPED) | run |
| 실행 기간 | `started_at ~ ended_at` (NULL = 진행중) | run |
| 유니버스 | `paper_runs.universe_snapshot` JSON 파싱 | run |
| 실현손익 | `SUM(pnl) FROM paper_trades WHERE run_id = :runId AND side = 'SELL'` | run |
| 수익률 | `실현손익 / SUM(qty*price) FROM paper_trades WHERE run_id=:runId AND side='BUY'` * 100 | run |
| 거래수 | `COUNT(*) FROM paper_trades WHERE run_id = :runId AND side = 'BUY'` (진입 횟수) | run |
| 최종 자산 | STOPPED: `equity FROM paper_equity_snapshots WHERE account_id=:acct AND ts <= ended_at ORDER BY ts DESC LIMIT 1`; RUNNING: 현재 account equity | account |

### 대시보드 탭 기여분 (3탭 중 Tab 1)

```
총 자산      = account.cash + SUM(position.qty * markPrice)  ← 계좌 전체 (account context)
가용 현금    = account.cash                                   ← 계좌 전체 (account context)
실현 손익    = SUM(pnl) WHERE run_id=:runId AND side='SELL'   ← run 기여분
미실현 손익  = SUM(qty * (markPrice - avgPrice)) WHERE symbol in run's open positions ← run 기여분
```

**"run의 오픈 포지션"** 파생 방법: `paper_trades WHERE run_id=:runId AND side='BUY'`에서 symbol set을 추출하고, 해당 symbol 중 `paper_trades WHERE run_id=:runId AND side='SELL'`에 없는 것들 → 그 symbol들을 `paper_positions`에서 조회. 이 방법은 인메모리 계산으로 충분히 빠르다(run당 거래 수가 수십~수백 건 수준).

### 성과 리포트 탭 기여분 (Tab 3)

- **이 실행만 (mode=SINGLE_RUN)**: equity curve = `paper_equity_snapshots WHERE account_id=:acct AND ts BETWEEN run.startedAt AND (run.endedAt OR now) ORDER BY ts ASC`. stats = run 기간 내 스냅샷 기반.
- **전략 전체 통합 (mode=RULE_AGGREGATE)**: equity curve = `paper_equity_snapshots WHERE account_id=:acct AND ts BETWEEN :from AND :to`. stats = 동기간. 복수 run의 realized PnL은 `SUM(pnl) WHERE rule_id=:ruleId AND traded_at BETWEEN :from AND :to`.

## Common Pitfalls

### Pitfall 1: PaperPosition UNIQUE 제약 파괴 시도

`paper_positions`에 run_id를 추가하고 UNIQUE를 (account_id, symbol, run_id)로 바꾸면 `PaperExecutor.executeBuy()`의 `findByAccountIdAndSymbol()` 가드가 무너진다. 두 룰이 같은 종목을 동시에 매수할 수 있게 되어 의도치 않은 중복 포지션이 생긴다.

**방지:** positions는 건드리지 않는다. "run의 오픈 포지션"은 paper_trades에서 파생한다(BUY without SELL in same run_id).

### Pitfall 2: 기간 필터 2-mode의 equity curve 불일치

"전략 전체 통합" 모드에서 equity curve를 단순히 ts 범위로 스냅샷 필터하면 계좌 전체의 곡선이 나온다(다른 룰들의 영향도 포함). 이를 "전략별 수익률"로 표현하면 오해를 일으킨다.

**방지:** 리포트 탭에 명확한 레이블: "이 실행만" 탭 = run 기간 내 account equity curve. "전략 전체 통합" 탭 = 선택 기간 내 account equity curve + "단일 계좌 공유 — 계좌 전체 자산 기준" 경고 문구.

### Pitfall 3: resolveActiveRunId()의 N+1 쿼리

PaperExecutor.execute()는 틱당 여러 종목에 대해 호출된다. BUY/SELL 신호가 여러 종목에서 동시 발생하면 resolveActiveRunId()가 룰당 여러 번 호출된다.

**방지:** evaluate 루프 내에서 runId를 룰별 1회만 조회하려면 LiveEvaluationService.evaluateRule()에서 runId를 미리 resolve하고 executor.execute()에 파라미터로 전달하는 것이 이상적이다. 단, OrderExecutorPort 인터페이스 변경을 수반하므로 Phase 6.9에서는 executor 내부 lookup으로 시작하고, 성능 이슈 발생 시 리팩토링한다.

### Pitfall 4: 토스 설정 라우트 완전 제거 금지

D4는 "PAPER에서 숨김"이지 "라우트 제거"가 아니다. `/trading/settings` 라우트를 제거하면 LIVE 모드 사용자도 접근 불가해진다. nav 항목만 조건부로 숨긴다.

### Pitfall 5: run 없는 상태의 start/stop (레이스 컨디션)

두 탭에서 동시에 start()를 호출하면 중복 RUNNING run이 생길 수 있다. `PaperLifecycleService.start()`에 이미 `ERR_LIFECYCLE_008` ("이미 RUNNING 상태인 룰") 가드가 있지만(`run_status='RUNNING'` 체크), paper_runs INSERT는 그 이후에 일어난다.

**방지:** start() 내에서 `@Transactional`이 이미 적용되어 있으므로 run_status 업데이트와 paper_runs INSERT를 같은 트랜잭션 안에서 실행하면 중복 생성이 방지된다.

### Pitfall 6: 구형 data(run_id=NULL)의 run-scoped 조회 혼입

PaperTradeRepository.findByRunId()를 단순히 WHERE run_id=:runId로 구현하면 NULL 행이 제외되어 OK. 하지만 "전략 전체 통합" 모드에서 rule_id + 기간 필터를 사용할 때 구형 NULL 데이터도 포함되는 것을 의도하므로, mode 파라미터에 따라 분기 쿼리가 필요하다.

## Architecture Patterns

### Recommended Wave Structure

**Wave 1 — Backend Foundation (DB + Domain)**
- V37 Flyway migration (paper_runs + paper_trades.run_id)
- `PaperRun.java` entity + `PaperRunRepository`
- `PaperTrade` 생성자에 runId 추가
- `PaperLifecycleService.start()/stop()` — run 생성/종료

**Wave 2 — Backend Execution Tagging + Services**
- `PaperExecutor` — resolveActiveRunId() + 2 edit points
- `PaperRunListService` — 리스트 집계 (realizedPnl, returnPct, tradeCount, finalEquity)
- `PaperRunContributionService` — 대시보드 탭 기여분, 포지션 파생

**Wave 3 — Backend REST API**
- `PaperRunListController` — GET /runs
- `PaperRunDetailController` — GET /runs/{id}/dashboard, /history, /report
- DTO: RunSummaryDto, RunDashboardDto (ReportDto 재사용)

**Wave 4 — Frontend IA + Nav**
- `TradingLayout.tsx` nav 재편 (3그룹, 토스 조건부 숨김)
- `router/index.tsx` routes 추가/제거
- `TradingRulesPage.tsx` 상단 시장·스케줄러 위젯 추가

**Wave 5 — Frontend 신규 페이지**
- `PaperRunsListPage.tsx` (Screen A)
- `PaperRunDetailPage.tsx` (Screen B, 3탭 + 기간 필터)

### Anti-Patterns to Avoid

- **run_id를 paper_positions에 추가하지 말 것**: UNIQUE 제약 및 executeBuy guard 파괴 위험.
- **account equity를 run별로 분리하지 말 것**: D5 확정(단일 계좌 공유). 총자산은 항상 account 전체.
- **PaperMonitorController를 삭제하지 말 것**: 전략 운영 화면 위젯이 이 엔드포인트를 소비한다.
- **TossSettingsPage 라우트를 삭제하지 말 것**: D4는 nav 숨김이지 라우트 제거 아님.

## Open Questions

1. **run 회차 표시**
   - 알려진 것: wireframe에 "#2회차" 레이블 표시.
   - 불명확: 회차 번호를 DB에 저장할지(paper_runs.run_index), 아니면 동일 rule_id의 started_at 순서로 실시간 계산할지.
   - 권장: 실시간 계산(`ROW_NUMBER() OVER (PARTITION BY rule_id ORDER BY started_at)`). 저장 시 동시성 문제 발생 가능.

2. **"수익률" 계산의 분모**
   - 알려진 것: 실현손익 / 투자원금 * 100 형태.
   - 불명확: 투자원금 = 최초 매수 원가 합계 vs account 초기 base_cash.
   - 권장: `SUM(qty*price) FROM paper_trades WHERE run_id=:runId AND side='BUY'`를 분모로 사용. 실제 투자한 금액 기준이므로 더 의미있음.

3. **equity snapshot 저장 빈도와 run-scoped 정확도**
   - 알려진 것: `LiveEvaluationService.saveEquitySnapshot()`가 매 틱 종료 후 저장(rule별 account에 대해).
   - 불명확: 복수 룰이 동시 실행 중일 때 틱당 여러 snapshots가 저장됨. run-scoped 기간 필터로 가져올 때 어떤 account-wide snapshot이 "이 run의" 것인지 구분 불가.
   - 권장: account-wide 곡선을 그대로 사용하고 run 기간(started_at~ended_at)으로 필터함을 UI에 명시. "이 기간 계좌 전체 자산 흐름" 레이블.
